## Arquitectura Modelo-Vista-Controlador (MVC)

El patrón MVC es una arquitectura de software que separa una aplicación en tres componentes lógicos principales, aislando la lógica de negocio de la interfaz de usuario. Esta separación de responsabilidades (_Separation of Concerns_) facilita el mantenimiento, la escalabilidad y el trabajo en equipo.

- **Modelo (Model):** Representa el dominio del problema. Gestiona la lógica de negocio, las reglas, los datos y el estado de la aplicación. Es independiente de la interfaz de usuario.
- **Vista (View):** Es la representación visual de los datos (la interfaz de usuario). Recibe la información del Modelo (generalmente a través del Controlador) y la renderiza para que el usuario pueda consumirla.
- **Controlador (Controller):** Actúa como el intermediario. Intercepta las interacciones del usuario desde la Vista, procesa esas peticiones invocando al Modelo para actualizar o recuperar datos, y finalmente determina qué Vista debe actualizarse y mostrarse.

---

## Cuándo usar MVC (y cuándo no)

La adopción del patrón MVC debe evaluarse según la naturaleza del proyecto.

### Cuándo SÍ utilizar MVC:

- **Aplicaciones web tradicionales:** Frameworks como Ruby on Rails, Django, Laravel o Spring Boot están construidos fuertemente sobre este patrón.
- **Equipos multidisciplinarios:** Cuando tienes desarrolladores frontend (enfocados en la Vista) y backend (enfocados en el Modelo/Controlador) trabajando en paralelo.
- **Proyectos con interfaces múltiples:** Si necesitas mostrar los mismos datos de múltiples formas (por ejemplo, una vista web, una vista móvil y un reporte en PDF), puedes reutilizar el Modelo y el Controlador, cambiando solo la Vista.

### Cuándo NO utilizar MVC:

- **Aplicaciones extremadamente pequeñas:** Para un script sencillo o una página de aterrizaje estática, la separación en tres capas genera sobreingeniería y complejidad innecesaria.
- **Aplicaciones web de una sola página (SPA) altamente interactivas:** En frameworks modernos (React, Vue, Angular), patrones como MVVM (Model-View-ViewModel) o arquitecturas basadas en componentes y flujo unidireccional (como Flux/Redux) suelen ser más eficientes para manejar el estado del lado del cliente.
- **Microservicios puros (Backend):** Si tu servicio solo expone una API REST y no renderiza ninguna interfaz gráfica, la capa de la "Vista" desaparece, convirtiéndose en una arquitectura más orientada a Servicios/Controlador-Modelo.

## Diagramas UML aplicables a MVC

Para documentar y entender un sistema basado en MVC, se utilizan principalmente los siguientes diagramas UML:

- **Diagrama de Clases (Estructural):** Se utiliza para modelar las entidades del sistema. Aquí se representan las clases del Modelo con sus atributos y métodos, las clases del Controlador y las clases/interfaces de la Vista, mostrando las relaciones de dependencia y asociación entre ellas.
- **Diagrama de Secuencia (Comportamiento):** Es el más importante para MVC. Muestra el flujo temporal de las llamadas. Ilustra cómo el "Actor" (usuario) interactúa con la Vista, cómo la Vista envía un evento al Controlador, cómo el Controlador consulta/actualiza el Modelo, y cómo el Modelo retorna los datos para que la Vista se actualice.
- **Diagrama de Paquetes (Arquitectónico):** Útil para visualizar cómo se agrupan lógicamente las clases en módulos separados (Paquete Modelos, Paquete Vistas, Paquete Controladores), demostrando la separación de responsabilidades.

## Plan de Pruebas para MVC

La separación en capas de MVC facilita enormemente la creación de un plan de pruebas estructurado. A continuación se detalla la estrategia recomendada:

| Componente           | Tipo de Prueba Recomendada       | Descripción y Alcance                                                                                                                                                                                                              |
| -------------------- | -------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Modelo**           | Pruebas Unitarias (Unit Tests)   | Al no depender de la UI, se prueban las reglas de negocio de forma aislada. Se verifica que los cálculos, las validaciones y el manejo de estado funcionen correctamente.                                                          |
| **Controlador**      | Pruebas Unitarias / Integración  | Se envían peticiones simuladas (_Mock requests_) al Controlador y se verifica que llame a los métodos correctos del Modelo y retorne la Vista o estructura de datos adecuada. Se suelen usar _Mocks_ para aislar la base de datos. |
| **Vista**            | Pruebas de UI / End-to-End (E2E) | Se evalúa la renderización del HTML/CSS y la interacción en el navegador. Se utilizan herramientas como Selenium o Cypress para simular clics y verificar que los elementos visuales respondan a los datos del Modelo.             |
| **Sistema Completo** | Pruebas de Integración           | Se prueba el ciclo de vida completo de la petición: desde que la ruta entra al Controlador, pasa por el Modelo (y la base de datos real o de pruebas) y retorna a la Vista.                                                        |

## Estructura de Carpetas en un IDE

Para mantener el código organizado, un proyecto MVC estándar adopta una estructura de directorios que refleja su arquitectura. Así se implementaría en tu IDE (por ejemplo, VS Code, IntelliJ o Eclipse) para una aplicación típica:

```text
/mi-proyecto-mvc
|-- /src
|   |-- /controllers       # Lógica de enrutamiento e intermediación
|   |   |-- ProductController.js
|   |   |-- UserController.js
|   |
|   |-- /models            # Lógica de negocio y esquemas de datos
|   |   |-- Product.js
|   |   |-- User.js
|   |
|   |-- /views             # Plantillas HTML, vistas renderizadas
|   |   |-- /products
|   |   |   |-- catalog.html
|   |   |   |-- detail.html
|   |   |-- /users
|   |   |   |-- profile.html
|   |
|   |-- /routes            # Definición de URLs apuntando a Controladores
|   |   |-- web.js
|   |
|   |-- /public            # Archivos estáticos
|   |   |-- /css
|   |   |-- /js
|   |   |-- /images
|   |
|   |-- /config            # Configuración de base de datos y entorno
|   |   |-- database.js
|   |
|   |-- app.js             # Punto de entrada de la aplicación
|
|-- /tests                 # Archivos de pruebas reflejando la estructura src
|   |-- /models
|   |-- /controllers
|
|-- package.json           # Dependencias del proyecto

```

> **Nota:** Esta organización garantiza que cualquier desarrollador que ingrese al proyecto sepa exactamente dónde encontrar la lógica de la base de datos, dónde modificar la interfaz gráfica y dónde alterar el flujo de las peticiones.

Este es un ejemplo de cómo implementar el patrón Modelo-Vista-Controlador en JavaScript puro (Vanilla JS).

## Ejemplo MVC

```javascript
// ==========================================
// 1. EL MODELO (Model)
// ==========================================
// Su única responsabilidad es gestionar los datos y el estado de la aplicación.
// No sabe nada sobre el HTML (Vista) ni sobre cómo se manejan las interacciones (Controlador).
class Model {
  constructor() {
    // Definimos el estado inicial de nuestro dato.
    this.message = "Hello World!!!!";
  }
}

// ==========================================
// 2. LA VISTA (View)
// ==========================================
// Su responsabilidad es mostrar la información al usuario (interfaz gráfica).
class View {
  // Recibe el modelo a través de inyección de dependencias.
  constructor(model) {
    // Guarda una referencia al modelo para poder leer sus datos posteriormente.
    this.model = model;
    // Captura el elemento del DOM (HTML) donde inyectaremos visualmente el mensaje.
    this.element = document.querySelector("#message");
  }

  // Este método toma los datos actuales del modelo y actualiza el HTML.
  render() {
    this.element.textContent = this.model.message;
  }
}

// ==========================================
// 3. EL CONTROLADOR (Controller)
// ==========================================
// Es el "director de orquesta". Conoce al Modelo y a la Vista.
// Recibe las acciones del usuario, actualiza el Modelo y le dice a la Vista que se actualice.
class Controller {
  // Recibe las instancias del modelo y la vista.
  constructor(model, view) {
    this.model = model;
    this.view = view;
    // Inmediatamente después de inicializar, fuerza el primer renderizado
    // para que la pantalla muestre el estado inicial ("Hello World!!!!").
    this.view.render();
  }

  // Método simulando una acción del usuario (ej. escribir en un input o hacer clic).
  setMessage(message) {
    // 1. Actualiza el estado (los datos) en el Modelo.
    this.model.message = message;
    // 2. Ordena a la Vista que se vuelva a dibujar (render) para reflejar el cambio.
    this.view.render();
  }
}

// ==========================================
// 4. INICIALIZACIÓN (Wiring/Bootstrap)
// ==========================================
// Se instancian las clases y se conectan entre sí para dar vida a la arquitectura.

const model = new Model(); // Creamos la fuente de datos.
const view = new View(model); // Creamos la vista y le pasamos el modelo para que sepa de dónde leer los datos.
const controller = new Controller(model, view); // Creamos el controlador, entregándole el control sobre el modelo y la vista.
```

---

## Explicación de la lógica

La lógica de este código se divide en dos fases principales: **La Inicialización** y **El Ciclo de Actualización**.

### 1. Fase de inicialización (Las últimas 3 líneas)

Cuando el script se ejecuta por primera vez, el sistema se ensambla a través de **Inyección de Dependencias** (pasar objetos a través del constructor):

- **El Modelo:** Nace con el texto `"Hello World!!!!"`.
- **La Vista:** Nace y se le entrega el Modelo. En este punto, la vista busca en el documento HTML un elemento con el ID `message` (por ejemplo, un `<div id="message"></div>`) y lo guarda en su memoria.
- **El Controlador:** Nace recibiendo tanto al modelo como a la vista. En su constructor, ejecuta `this.view.render()`. Esto hace que el texto `"Hello World!!!!"` aparezca inmediatamente en la pantalla del usuario.

### 2. Ciclo de actualización (El flujo de datos)

El verdadero poder de MVC se demuestra cuando algo necesita cambiar. Si en otra parte de tu código, o mediante un evento de un botón, llamas al método `setMessage` del controlador:

```javascript
controller.setMessage("¡Hola desde el Arquitecto de Software!");
```

La lógica ejecutará exactamente este flujo unidireccional:

- **Paso 1 (Controlador):** El Controlador intercepta tu petición y ejecuta `this.model.message = message;`.
- **Paso 2 (Modelo):** El Modelo cambia internamente su estado. Ya no dice "Hello World", ahora guarda el nuevo texto. _Nota: El modelo no hace nada más, su trabajo termina aquí._
- **Paso 3 (Controlador de nuevo):** El Controlador sabe que el dato cambió, por lo que invoca `this.view.render();`.
- **Paso 4 (Vista):** La Vista obedece al Controlador, lee el dato más reciente directamente de `this.model.message` y reemplaza el texto en el DOM del navegador.

> **Nota:** Este enfoque es una variante común donde **la Vista conoce al Modelo** (tiene acceso de lectura a sus propiedades). Esto es válido y muy usado. Existe otra variante más estricta (_Passive View_) donde la Vista no conoce al Modelo en absoluto, y es el Controlador quien extrae el dato del Modelo y se lo inyecta a la Vista (ej. `this.view.render(this.model.message)`). Ambas son correctas dependiendo de qué tan acoplados se desea que estén los componentes.
