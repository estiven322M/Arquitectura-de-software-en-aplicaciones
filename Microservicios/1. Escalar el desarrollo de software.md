# Microservicios para Escalar el Desarrollo de Software

## Introducción

Uno de los argumentos más sólidos a favor de una arquitectura basada en microservicios es su capacidad para escalar el desarrollo organizacional. Mientras que muchas tecnologías permiten escalar la infraestructura mediante la adición de servidores, los microservicios permiten escalar el trabajo de los equipos de desarrollo.

En sistemas empresariales de gran tamaño, donde participan decenas o incluso cientos de desarrolladores, el principal problema no suele ser técnico sino organizacional. A medida que aumenta el tamaño de un sistema monolítico, aumenta también la cantidad de coordinación necesaria entre equipos, la complejidad de las dependencias internas y el riesgo de conflictos durante el desarrollo.

Los microservicios abordan este problema mediante la división del sistema en unidades autónomas, cada una responsable de una capacidad de negocio específica.

---

# División del sistema en unidades independientes

En una arquitectura monolítica tradicional, todos los módulos forman parte de una única aplicación desplegable. Aunque existan separaciones lógicas en el código, todos los equipos trabajan sobre la misma base de código.

Esto genera problemas como:

- Conflictos frecuentes entre desarrolladores.
- Dependencias entre equipos.
- Ciclos de despliegue complejos.
- Mayor riesgo de introducir errores en componentes ajenos.

Por el contrario, en una arquitectura de microservicios:

- Cada servicio posee su propio código.
- Cada servicio puede tener su propio repositorio.
- Cada servicio puede ser desplegado de forma independiente.
- Cada servicio posee una interfaz claramente definida.

Por ejemplo:

```text
Sistema de Comercio Electrónico

├── Servicio de Clientes
├── Servicio de Productos
├── Servicio de Inventario
├── Servicio de Pagos
├── Servicio de Facturación
└── Servicio de Notificaciones
```

Cada uno puede ser desarrollado por equipos distintos sin necesidad de modificar directamente el código de los demás.

---

# Independencia tecnológica

Una de las características más importantes de los microservicios es la independencia tecnológica.

Mientras un servicio exponga correctamente su interfaz pública, el resto del sistema no necesita conocer su implementación interna.

Supóngase un servicio de productos:

```http
GET /productos/{id}
```

Los consumidores únicamente conocen:

- La URL.
- El formato de la petición.
- El formato de la respuesta.

No conocen:

- El lenguaje de programación utilizado.
- La estructura interna del código.
- La base de datos empleada.
- Las bibliotecas utilizadas.

Por esta razón, un equipo podría desarrollar un servicio utilizando:

- Java con Spring Boot.
- Python con FastAPI.
- Go.
- Node.js.
- .NET.

Siempre que la interfaz permanezca estable.

Esta independencia tecnológica se conoce como **Technology Heterogeneity**.

---

# Uso de contenedores para aislamiento

Cuando los microservicios se distribuyen mediante contenedores Docker, el nivel de independencia aumenta aún más.

Un contenedor encapsula:

- Aplicación.
- Dependencias.
- Bibliotecas.
- Configuración.
- Runtime.

Por ejemplo:

```text
Contenedor A
├── Java 21
├── Spring Boot
└── PostgreSQL Driver

Contenedor B
├── Python 3.12
├── FastAPI
└── SQLAlchemy

Contenedor C
├── Node.js
├── Express
└── MongoDB Driver
```

Desde el punto de vista de otros servicios, únicamente importa que la API funcione correctamente.

El interior del contenedor se convierte en una caja negra.

---

# Despliegues independientes

Otro beneficio fundamental es la independencia de despliegue.

En un monolito:

```text
Cambio pequeño
      ↓
Recompilar aplicación completa
      ↓
Probar aplicación completa
      ↓
Desplegar aplicación completa
```

En microservicios:

```text
Cambio en Servicio de Inventario
          ↓
Compilar Inventario
          ↓
Probar Inventario
          ↓
Desplegar Inventario
```

Los demás servicios permanecen intactos.

Esta capacidad permite:

- Entregas continuas.
- Despliegues frecuentes.
- Menor riesgo operacional.
- Mayor velocidad de innovación.

---

# Sustitución de sistemas legados

Otro uso importante de los microservicios es la modernización de sistemas heredados o _legacy systems_.

Un sistema legado suele presentar problemas como:

## Código mal estructurado

Con frecuencia los sistemas antiguos evolucionaron durante años o décadas sin una arquitectura consistente.

Como consecuencia aparecen:

- Acoplamiento excesivo.
- Duplicación de código.
- Dependencias ocultas.

## Ausencia de pruebas automatizadas

Muchos sistemas heredados fueron construidos antes de que las pruebas automatizadas fueran una práctica común.

Por ello:

- Cada modificación genera incertidumbre.
- Los defectos son difíciles de detectar.
- Los tiempos de validación aumentan.

## Tecnologías obsoletas

Es frecuente encontrar:

- Frameworks abandonados.
- Versiones antiguas de lenguajes.
- Bases de datos descontinuadas.

Esto dificulta:

- La contratación de desarrolladores.
- La incorporación de nuevas funcionalidades.
- La seguridad del sistema.

---

# Estrategia de reemplazo progresivo

Los microservicios permiten modernizar sistemas sin necesidad de reescribirlos completamente.

Esta estrategia suele conocerse como **Strangler Pattern**.

El enfoque consiste en reemplazar gradualmente funcionalidades del sistema antiguo.

Ejemplo inicial:

```text
Sistema Legado
│
├── Gestión de Clientes
├── Gestión de Productos
├── Facturación
└── Inventario
```

Primera etapa:

```text
Sistema Legado
│
├── Gestión de Clientes
├── Facturación
└── Inventario

Nuevo Microservicio
└── Productos
```

Etapa posterior:

```text
Sistema Legado
│
├── Facturación
└── Inventario

Microservicios
├── Clientes
└── Productos
```

Y así sucesivamente hasta retirar completamente el sistema antiguo.

# Mecanismos de integración

Durante la transición ambos mundos deben coexistir.

Para ello pueden emplearse mecanismos como:

## APIs REST

```text
Sistema Legado
      ↓
REST API
      ↓
Microservicio
```

## Mensajería

Utilizando tecnologías como:

- Apache Kafka
- RabbitMQ
- ActiveMQ

Ejemplo:

```text
Evento Cliente Creado
        ↓
Broker de Mensajes
        ↓
Microservicio
```

## Replicación de datos

Se sincronizan datos entre ambos sistemas.

```text
Base de Datos Legada
          ↓
 Replicación
          ↓
Base de Datos Nueva
```

## Integración a nivel de interfaz

El usuario puede interactuar con aplicaciones antiguas y nuevas de manera transparente.

Por ejemplo mediante:

- Portales integrados.
- Single Sign-On.
- Frontends compuestos.

# Desarrollo sostenible a largo plazo

Una promesa central de los microservicios es la sostenibilidad arquitectónica.

La idea es que el sistema continúe siendo mantenible incluso después de muchos años de evolución.

---

# Reemplazabilidad de los microservicios

Un concepto fundamental es la reemplazabilidad.

Supóngase que un servicio se vuelve difícil de mantener:

```text
Servicio de Facturación
```

Problemas:

- Código complejo.
- Tecnología obsoleta.
- Bajo rendimiento.

En un monolito:

```text
Modificar componente
       ↓
Impacto potencial sobre todo el sistema
```

En microservicios:

```text
Reescribir servicio
       ↓
Mantener misma interfaz
       ↓
Sustituir implementación
```

El resto del ecosistema permanece funcional.

Esto permite:

- Renovación tecnológica continua.
- Reducción de deuda técnica.
- Evolución gradual de la arquitectura.

---

# Gestión de dependencias

La reemplazabilidad solamente es posible si las dependencias están correctamente controladas.

Un microservicio altamente dependiente de otros puede convertirse en un punto crítico.

Por ejemplo:

```text
Servicio A
   ↓
Servicio B
   ↓
Servicio C
   ↓
Servicio D
```

Si el Servicio B cambia:

```text
Afecta A
Afecta C
Afecta D
```

Por esta razón se recomienda:

- Minimizar dependencias.
- Mantener contratos estables.
- Aplicar versionado de APIs.
- Evitar llamadas innecesarias entre servicios.

---

# Problemas de las arquitecturas clásicas

En arquitecturas tradicionales aparecen frecuentemente violaciones arquitectónicas.

Supóngase:

```text
Módulo Ventas
Módulo Inventario
Módulo Facturación
```

La arquitectura establece:

```text
Ventas → Inventario
Ventas → Facturación

Facturación NO → Inventario
```

Sin embargo, un desarrollador puede introducir accidentalmente:

```text
Facturación → Inventario
```

Debido a que:

- Todo el código reside en el mismo proyecto.
- Los límites arquitectónicos son débiles.
- No existe aislamiento físico.

Con el tiempo aparecen numerosas dependencias no planificadas.

La arquitectura original se degrada progresivamente.

Este fenómeno suele denominarse **Architectural Erosion** (Erosión Arquitectónica).

---

# Límites claros en microservicios

Los microservicios introducen límites explícitos.

Un servicio sólo puede interactuar con otro mediante su interfaz pública.

Por ejemplo:

```text
Servicio Clientes
        ↓
REST API
        ↓
Servicio Pedidos
```

No existe acceso directo al código interno.

Tampoco existe acceso directo a:

- Clases.
- Métodos.
- Objetos.
- Componentes internos.

Todo acceso debe pasar por la interfaz definida.

---

# Firewalls arquitectónicos

Las interfaces de microservicios actúan como verdaderos **firewalls arquitectónicos**.

Un firewall arquitectónico es un mecanismo que impide la creación accidental de dependencias indebidas.

Ejemplo:

```text
Microservicio A
        ↓
     API
        ↓
Microservicio B
```

Si se desea introducir una nueva dependencia:

1. Debe modificarse el contrato.
2. Debe publicarse una nueva API.
3. Deben adaptarse los consumidores.

La dependencia deja de ser accidental y pasa a ser una decisión explícita.

---

# Calidad arquitectónica a largo plazo

La consecuencia más importante de todo lo anterior es que los microservicios ayudan a preservar la calidad arquitectónica durante largos períodos de tiempo.

Esto ocurre porque:

- Los límites son visibles.
- Los contratos son explícitos.
- Los despliegues son independientes.
- Las dependencias son controladas.
- Los servicios pueden reemplazarse individualmente.

En consecuencia, una arquitectura basada en microservicios no solamente busca resolver problemas técnicos inmediatos. Su objetivo principal es permitir que sistemas complejos continúen evolucionando durante años o décadas sin caer en la degradación estructural típica de muchos sistemas monolíticos de gran tamaño.

La verdadera ventaja estratégica de los microservicios no radica únicamente en la escalabilidad técnica, sino en la capacidad de mantener la independencia organizacional, tecnológica y arquitectónica a lo largo del tiempo.
